# NEIS 교과평가 자동입력기 — C# 개발 로드맵

작성일: 2026-07-07
전략: **위험이 낮고 테스트 가능한 안쪽 계층(Core/Excel)부터 → 브라우저 엔진 → GUI → 미해결 이슈 → 확장**
원칙: 각 단계는 독립적으로 검증 가능해야 하고, 프로토타입에서 **이미 검증된 알고리즘을 바꾸지 않는다**.

전체 예상 규모: 개인 개발 기준 약 **5~7일** (엔진 이식이 가장 큰 덩어리).

## Phase 3 — 콤보 조작 & 입력 엔진  (1.5일)

**목표:** 실제 단계 입력 + 재검증 루프. **여기가 자동화의 심장.**

- [ ] `ComboBoxDriver.OpenAndPickAsync` (§3.4·3.5)
      - **네이티브 클릭** (`ClickAsync`) 으로 콤보 열기 — JS 클릭/키/ActionChains 전부 금지(§3.7)
      - `.cl-global-aside` 옵션 폴링 (2.5s / 0.15s), 텍스트 일치 옵션 클릭, NBSP 제거
- [ ] `GetFreshComboAsync` / `EnsureRowVisibleAsync` — 조작 직전 참조 재획득 (§3.6·4.3)
- [ ] `SetGradeAsync` — 멱등성 + 입력 + 재검증 + 1회 재시도 (설계도 §3.3)
- [ ] `RunSubjectAsync` — §4.1 전체 흐름 조립 (과목검증 → 행지도 → todo → 순회 → 리포트)
- [ ] **dry-run 우선 검증** → 그다음 실제 1~2행만 입력 테스트 → 전체
- [ ] 안전장치 전부 배선 (설계도 §5): 과목 불일치 거부, 화이트리스트, 저장 미자동화

**완료 기준:** 국어 dry-run 계획 정확 + 실입력 50/51 성공 + 재실행 시 "이미 설정됨" 멱등 동작.

## Phase 5 — ★ 마지막 행 미렌더링 해결  (0.5~1일, 최우선 이슈)

**목표:** 문서 §8 의 유일한 미해결 이슈 해결. Playwright 채택의 본래 목적.

해결 순서(§8.4 권장순, 위에서부터 시도):

- [ ] ④ 최후: 미처리 행을 리포트 + 수동 입력 안내 (현재 동작 유지)

**완료 기준:** 51/51 전 행 입력 성공. 안 되면 원인 문서화 + ④ 폴백 확정.

**Phase 1·3에 스며드는 교차 작업** — Core 완성 직후 착수하면 이후 단계가 척도 기반으로 굴러간다.

- [ ] `GradeLevel` / `GradeScale` / `IScaleStore`(scales.json) 모델 (설계도 §9.2)
- [ ] 내장 프리셋: 잘함·보통·노력요함 / 상·중·하 / 4단계 / 5단계

**완료 기준:** 5단계 척도 엑셀을 넣어도 파싱·dry-run 정상, 나이스 옵션에 없는 등급은 안전 skip.

## 개발 순서 원칙 (왜 이 순서인가)

1. **안쪽부터**: Core/Excel 은 브라우저 없이 테스트되므로 로직 신뢰도를 먼저 고정.
2. **읽기 → 쓰기**: Phase 2(읽기)로 화면 파싱을 검증한 뒤 Phase 3(입력)으로. 파싱이 틀리면 입력도 틀린다.
3. **엔진 → GUI**: 엔진을 CLI/테스트로 먼저 완성(§9 프로토타입도 fill_korean3.py CLI가 선행). GUI는 껍데기.
4. **검증된 알고리즘 보존**: §3.7 "동작하지 않는 방법 목록"을 절대 재시도하지 않는다.
```
