작성: 2026-07-17 (v1.3.1 직후) · 상태 표기: ⬜ 대기 · 🔄 진행중 · ✅ 완료
배경: v1.3.x 에서 기능이 빠르게 자라며(4,800→8,650라인) 구조 부채가 쌓임.

### 1.4 배포 형태 검토 (요청사항) — **실측 결과: 현행 유지가 정답**

세 가지 레이아웃을 같은 코드로 빌드해 창이 뜰 때까지 실측 (SSD, 2·3회차 평균 / 1회차):

| 레이아웃 | 디스크 크기 | 파일 수 | 시작 속도(워밍업 후) | 최초 1회 실행 |
|---|---|---|---|---|
| **압축 단일 exe (현행)** | **101 MB** | 3항목 | 1.68초 | **1.8초** |
| 비압축 단일 exe | 247 MB | 3항목 | 1.43초 | 5.8초 (추출·검사) |
| 다중 파일 (구방식) | 259 MB | 396개 | 1.68초 | 2.6초 |

- 워밍업 후 차이는 **최대 0.25초 — 체감 불가**. 압축 해제 비용은 최신 CPU 에서 무시 수준이고, 느린 학교 PC(HDD)에서는 오히려 "파일 396개 열기(IO)"가 더 불리해 현행이 유리할 가능성이 큼
- 비압축 단일은 워밍업 후 가장 빠르지만 디스크 2.4배 + 최초 실행 5.8초로 손해가 더 큼
- "작은 exe + 소수 DLL" (framework-dependent, 총 ~20MB) 은 사용자 .NET 설치가 필요해 배제 (기존 결정)
- **결론: 현행(압축 단일 exe)이 크기·정돈·시작 속도 모두에서 최적. 변경 불필요.**
- 덤: 시작 체감을 더 좋게 하고 싶다면 레이아웃이 아니라 **스플래시 화면**(즉시 표시)이 정답 — 선택 항목으로 R5 에 포함

## 2. 리팩토링 플랜 (단계별 — 각 단계 독립 커밋·검증)

### R1 — 순수 로직을 Core 로 내리고 테스트 씌우기 ✅ `25b430f` (효과 최대·위험 최소, 최우선)
- `Core/SheetSynchronizer` 신설: BuildSheetFromRoster·BuildAreaRenameMap·Carry·SheetShapeEquals 를 MainViewModel 에서 이동 (S2)
- `Core/Matching/SimilaritySuggester` 신설: MatchPreviewViewModel 의 유사도·기본값 선택(이름 일치→남는 영역 순서) 로직 이동, NameNormalizer 재사용 (B6)
- 각각 단위 테스트 추가 (개명 보존·명단 증감·기본값 선택 시나리오) — **오늘 버그났던 지점을 전부 테스트 아래로**
- 예상: 테스트 75 → 90+

### R2 — 전과목 오케스트레이터 통합 ✅ `dc88155` (S3)
- `App/Services/BatchUploadRunner` 신설: "과목 목록 → 과목마다 전환·실행·검증·저장·중단 규칙·요약" 공통 루프 1벌
- 등급/서술문은 `Func<subject, Task<RunResult>>` 실행자만 주입
- MainViewModel·GeneratorViewModel 에서 각 ~80줄 삭제

### R3 — MainViewModel 분해 🔄 부분 완료 `a7c31e9` (S1)
- ✅ 성취기준 패널 구성 로직 → Core/CriteriaPanelBuilder (순수화)
- ✅ `Sheet` → `Snapshot()` 개명 — 호출 시점 고정 의도 명시 (B1)
- ✅ 구 양식 커맨드·DataPrepWindow 제거로 자연 감량 (R2 에서)
- ⬜ WorkspaceService 추출 (파일 로드/자동저장/최근파일/복원/내보내기) —
  Subjects 컬렉션과 강결합이라 별도 세션에서 신중히. 현재 MainViewModel ~800라인

### R4 — 코드비하인드 정리 ⬜ (S4)
- 성적표 상호작용(일괄 지정·복붙·행/열 선택)을 attached behavior 또는 GradeGridController 로 추출
- MainWindow.xaml.cs 목표 377 → 150라인 이하 (창 제어·컬럼 생성만)

### R5 — 소청소 + 안정화 🔄 대부분 완료 (B2~B5, E1~E3)
- ✅ `Diag.Swallow` 훅 (자동연결 루프 적용 — diag.txt, 세션당 중복 억제)
- ✅ OpenHelp async void 제거, AutoConnectLoop 취소 토큰
- ✅ DataPrepWindow·양식 커맨드·MaxNarrativeBytes 죽은 코드 제거
- ✅ 테스트 TFM net8.0-windows (CA1416 경고 5건 해소 — 경고 0 복귀)
- ❌ NarrativeStore 저장 디바운스 — **검토 후 기각**: 테스트가 '즉시 보존 보장' 회귀를
  잡아냄. 데이터 안전 > 쓰기 절약 (코드에 결정 주석)
- ❌ 스플래시 화면 — **실측 후 불필요 판정**: 창까지 1.3~1.7초로 스플래시 권장 기준(3초+)에
  한참 못 미침. 이 속도에선 깜빡임이 되어 역효과. (HDD 최초 실행이 느린 경우는 exe 로드
  자체가 병목이라 스플래시로도 해결 불가)
- ⬜ 잔여: 나머지 빈 catch 에 Diag 훅 점진 적용, HttpClient 팩토리

### 진행 순서와 이유
**R1 → R2 → R3 → R4 → R5.** R1 이 안전망(테스트)을 먼저 깔고, R2·R3 의 큰 이동은 그 안전망 위에서. R4·R5 는 독립적이라 언제든.
각 단계 완료 기준: 빌드 경고 0 + 테스트 전부 통과 + 앱 부팅 + (R2·R3 은) 성적 편집→저장→재시작 복원 수동 확인.
나이스 실기기 검증이 필요한 변경은 없음 — 엔진(Automation)은 이번 리팩토링에서 건드리지 않는다.
