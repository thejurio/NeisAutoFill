# 전담 모드 설계·개발 로드맵 (F9)

전담 교사가 **여러 학년·반의 특정 과목**을 가르치는 상황을 지원하기 위한 설계.
담임 모드(현재)는 그대로 두고, 전담 모드를 **별도 데이터 구조**로 얹는다.

> 상태: **설계 확정용 문서 (아직 미구현)**. Phase A(경로 인프라)만 커밋됨.
> 이 문서는 나중에 그대로 AI 에게 주고 "이 로드맵대로 M2 부터 개발해줘" 라고 시킬 수 있게 자세히 적었다.
> 작성: 2026-07-18

---

## 1. 문제 — 담임과 전담은 데이터 모양이 다르다

| | 담임 (현재) | 전담 |
|---|---|---|
| 명단 | 우리 반 하나 | **반마다** 다름 (3-1, 3-2, 4-1…) |
| 평가계획 | 과목별 (우리 반 전 과목) | **(학년·과목)마다** 다름. 같은 학년·과목이면 여러 반이 공유 |
| 성적·서술문 | 과목별 (한 반) | **(학년·반·과목) 조합마다** |

핵심 통찰: 전담의 데이터는 **세 개의 축**으로 정규화된다.
- **명단** = `(학년, 반)` 의 함수
- **평가계획** = `(학년, 과목)` 의 함수 — 반은 무관 (3-1 영어와 3-2 영어의 평가계획은 같다)
- **작업(성적·서술문)** = `(학년, 반, 과목)` = 명단 × 평가계획의 교차점

→ 담임 구조("한 파일에 과목 시트들")에 억지로 담으면 명단·계획이 반 수만큼 중복된다.

---

## 2. 설계 결정 — 분리 저장(정규화). 담임 파일과 전담 파일은 완전히 따로

### 왜 분리 저장인가 (통째 복사 대신)
- **현실 반영**: 3-1·3-2가 같은 "3학년 영어 평가계획"을 공유하는 게 실제 모습. 통째 복사면 같은 계획을 반 수만큼 다시 입력해야 한다.
- **AI 통합 분석과 자연스럽게 연결**: "평가계획을 한 파일에 다 넣고 AI가 학년·과목별로 분리" 요구(§5)가 곧 `(학년·과목)` 단위 저장으로 귀결된다.
- **수정 편함**: 계획 하나 고치면 그 학년 전 반에 반영. 명단도 반별 1회 입력으로 그 반의 전 과목이 공유.
- **단점**(감수): 구조가 복잡하고 개발이 크다. 하지만 실사용 가치가 훨씬 크다.

### 폴더 레이아웃
```
문서\NeisAutoFill\
 ├ 성적.xlsx  평가계획서.xlsx  서술문.xlsx        ← 담임(현재). 전담 켜기 전까지 안 건드림
 └ 전담\
    ├ 명단\        {학년}-{반}.xlsx               ← 반별 학생명단 (번호·이름)   예: 3-1.xlsx
    ├ 평가계획\    {학년}_{과목}.xlsx             ← (학년·과목)별 평가계획       예: 3_영어.xlsx
    └ 작업\        {학년}-{반}_{과목}\            ← 조합별 결과                 예: 3-1_영어\
                     성적.xlsx  서술문.xlsx
```
- 내부 데이터(narratives.json·state.json)는 `%AppData%\NeisAutoFill\profiles\{조합}\` (Phase A 의 `ProfilePaths.DataFile` 이미 지원).
- **담임 무영향**: `전담\` 폴더는 전담 모드를 켜야만 생긴다. 담임 사용자는 기존 경로 그대로 (Phase A 의 `ProfilePaths.IsDefault`).

## 3. 도메인 모델 (3층)

```
[등록 정보]  전담이 담당하는 것
  · 학년별 반 목록      예: 3학년→[1,2], 4학년→[1]
  · 담당 과목           예: 영어(3·4학년), 과학(4학년)  ← 학년마다 과목이 다를 수 있음
      → 담당 조합 = Σ (학년, 반, 과목)  자동 생성 또는 직접 등록

[자료]
  · 명단[(학년,반)]        ← 반별
  · 평가계획[(학년,과목)]   ← 학년·과목별

[작업]  = 현재 선택한 (학년,반,과목) 조합
  · 명단[(학년,반)] + 평가계획[(학년,과목)] 를 합쳐 성적표 구성
  · 성적·서술문은 이 조합 폴더에 저장
```

### 등록 정보를 어떻게 받나 (UI 안)
- (A) **행렬형**: 학년·반 체크 + 과목 체크 → 교차로 조합 자동 생성. 빠르지만 "3학년은 영어만, 4학년은 과학만" 같은 비대칭이 어려움.
- (B) **조합 직접 등록**: `(학년, 반, 과목)` 을 한 줄씩 추가. 유연하지만 손이 많이 감.
- **권장: (A) + 예외 편집**. 학년별로 "이 학년에서 담당할 과목"과 "반"을 고르면 조합이 생기고, 필요 시 개별 조합을 빼거나 추가. → M3 에서 UX 상세.

## 4. 담임 vs 전담 모드 전환

- 설정 › 일반에 **담임 / 전담** 라디오 (Phase B 에서 시도했던 UI 재활용, 단 세트 모델이 아니라 위 3층 모델로).
- 전담으로 바꾸면: 등록 UI(학년·반·과목) 노출 → 조합 목록 생성 → 현재 조합 선택.
- **조합 전환은 재시작**으로 반영 (싱글턴 재구성 회피 — Phase A 방침). `AppReset.RestartApp` 재사용.
- 설정 저장 위치: `profiles.json` 확장 (mode, 등록 정보, 현재 조합). 공용(프로필 무관).

## 5. AI 통합 평가계획 분석 (학년·과목 자동 분리)

전담은 담당하는 여러 (학년·과목)의 평가계획을 **한 파일(또는 몇 개)에 몰아넣고**, AI 가 알아서 쪼개길 원한다.

### 현재 (담임, 2단계 파싱 — GasPlanImporter)
1. `listPlanSubjects`: 문서에서 **과목명** 목록 발견
2. `parsePlan(onlySubject)`: 과목별로 나눠 완전 추출

### 확장 (전담 — 학년 축 추가)
1. `listPlanUnits`(신규): 문서에서 **(학년, 과목)** 쌍 목록 발견
   - 프롬프트에 "학년"도 인식하도록. 예: "3학년 영어", "4학년 과학"
   - 문서에 학년 표기가 없으면(한 학년치만) → 사용자가 고른 학년으로 귀속하거나 담임식 폴백
2. `parsePlanUnit(onlyGrade, onlySubject)`(확장): (학년·과목)별 완전 추출
3. 결과를 `평가계획\{학년}_{과목}.xlsx` 로 **각각 저장** → 해당 반들이 참조

### 구현 메모
- `GasPlanImporter` 의 `ListSubjectsAsync`/`ParseSubjectAsync` 에 `grade` 파라미터 추가, 또는 "단위(unit)" 개념으로 일반화.
- code.gs 의 `listPlanSubjects`/`parsePlanDocument` 프롬프트에 학년 인식 지시 추가. (셀렉터와 달리 프롬프트는 GAS 재배포만으로 갱신 가능)
- **하위호환**: 담임 경로(과목만)는 그대로 두고, 전담 경로에서만 학년 축을 쓴다. 또는 grade=null 이면 현재 동작.

## 6. 개발 마일스톤 (단계별 — 각 단계 독립 커밋·검증)

| M | 내용 | 위험 | 실기기 |
|---|---|---|---|
| **M0** ✅ | 경로 인프라 (ProfilePaths·AppPaths·ProfileStore). 담임 기본=기존 경로 (Phase A, 완료) | 낮음(동작무변화) | 불필요 |
| **M1** ✅ | 데이터 계층: 도메인 모델(ClassRef·TeachingUnit)·경로 규칙(SubjectModePaths)·조합 조립(BuildUnitSheet). 순수 로직 테스트 15개 | 중 | 불필요 |
| **M2** ✅ | 등록 모델: SubjectAssignment(학년→반들·과목들→조합 생성, 제외·비대칭·중복정리), ProfileStore 3층 재작성. 순수 로직 테스트 7개 | 중 | 불필요 |
| **M3** | 전담 UI: 설정에 모드+등록(학년·반·과목), 조합 선택·전환(재시작). 담임 무영향 확인 | 중(UI) | 앱 확인 |
| **M4** | AI 통합 분석: (학년·과목) 분리 인식(listPlanUnits/parsePlanUnit) + code.gs 프롬프트. 결과를 학년과목별 저장 | 중 | GAS 재배포 |
| **M5** | 작업 흐름 연결: 조합 선택→명단+계획 합쳐 성적표→생성→나이스 입력. 나이스 조회도 (학년·반·과목) | 높음 | **실기기 필수** |
| **M6** | 마감: 백업/복원·초기화가 전담 구조 인지, 문서화, 실사용 검증 | 중 | 앱+실기기 |

### 진행 원칙
- **M1·M2 는 순수 로직 위주** → 테스트로 덮고 안전하게. 여기서 데이터 모델을 확정.
- **M3 이후 UI·엔진**은 실기기 확인 전제. 특히 **M5(나이스 조회·입력)** 는 전담이 반·과목을 바꿔가며 조회하므로 실기기 검증이 반드시 필요.
- 각 M 마다 **담임 모드 회귀 없음**을 확인(담임은 M0 경로로 계속 동작).

## 8. 리스크 · 미결정

- **나이스 조회 단위**: 전담이 반·과목을 바꿀 때 나이스 화면 전환이 담임과 다른지 실기기로 확인 필요(M5).
- **학년 표기 인식**: 평가계획 문서에 학년이 명확히 안 적혀 있으면 AI 분리가 애매. 폴백(사용자 지정) 필요(M4).
- **조합 폭발**: 담당이 많으면 조합이 수십 개. UI 가 감당하는지(목록·검색)(M3).
- **명단·계획 수정 파급**: 계획 하나 고치면 그 학년 전 반의 성적표에 영향 — 재계산·보존 규칙(M1, 담임의 "영역 개명 보존" 로직 참고).
- **창의적 체험활동·행동특성**(향후): 담임이 프로필 안에서 추가 입력 영역으로. 이 구조가 그릇이 됨 — `ProfilePaths.DataFile` 이 파일명 파라미터라 같은 폴더에 `창체.json` 등 공존 가능. 전담 설계와 독립적.
