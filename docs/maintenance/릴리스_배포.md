# 릴리스 · 배포 절차

새 버전을 내보내는 표준 절차. **버전 → 빌드 → 게시**를 순서대로.
매번 코드를 뒤지지 않도록 여기만 보고 진행한다.

---

## 0. 사전 조건

- .NET SDK는 **사용자 로컬**에 설치돼 있다(시스템 PATH 아님). 모든 `dotnet` 명령 전에 환경변수를 잡는다:
  ```powershell
  $env:DOTNET_ROOT="$env:LOCALAPPDATA\Microsoft\dotnet"; $env:PATH="$env:DOTNET_ROOT;$env:PATH"
  ```
- GitHub는 `gh` CLI(계정 `thejurio`)로 조작한다. 저장소: https://github.com/thejurio/NeisAutoFill (public)

---

## 1. 버전 올리기 (한 곳만)

버전의 **단일 원본**은 다음 한 줄뿐이다. 나머지(AssemblyVersion 등)는 빌드가 자동 생성한다.

```
src/NeisAutoFill.App/NeisAutoFill.App.csproj
  <Version>1.4.1</Version>
```

- 이 값이 곧 자동업데이트 비교 기준이자 GitHub 릴리스 태그(`v1.4.1`)와 일치해야 한다.
- **버전 규칙(SemVer 준용)**
  | 변경 성격 | 예 | 올림 |
  |---|---|---|
  | 버그 수정 · 작은 기능 추가 | 명단 삭제 버그, 초기화 버튼 | patch (1.4.0 → 1.4.1) |
  | 기능 묶음 · UX 개편 | UI 전면 개편, AI 계획 가져오기 | minor (1.3.1 → 1.4.0) |
  | 호환 깨지는 대변경 | (아직 없음) | major |

## 2. 빌드 · 테스트 (게시 전 필수)

```powershell
dotnet build --nologo         # 경고 0 · 오류 0 확인
dotnet test  --nologo         # 전 항목 통과 확인 (현재 91개)
```

> 이 build+test 는 **GitHub Actions CI**(`.github/workflows/ci.yml`)가 매 푸시·PR 마다 windows 러너에서 자동 실행한다.
> README 의 CI 배지로 상태를 볼 수 있다. 로컬 확인은 여전히 권장(빠른 피드백).
> CI 는 `GasSecret.cs`(gitignore) 없이 돌아가므로 시크릿을 요구하지 않는다(빈 시크릿으로 컴파일 — 빌드·테스트엔 무관).

# 자동 릴리스 (권장) — 태그 push 하나로 끝

`.github/workflows/release.yml` 가 **태그 push 를 감지해 검증·빌드·게시**를 자동으로 한다.
아래 §3~§5(수동 게시)는 CI 를 못 쓸 때의 폴백이다.

### 사전 준비 (1회만)
저장소 **Settings → Secrets and variables → Actions** 에 시크릿 추가:
- `GAS_SECRET` = `code.gs` 의 `V2_SECRET` 과 동일한 값 (서명용). CI 가 빌드 때 `GasSecret.cs` 를 이 값으로 생성한다.

### 릴리스 절차
```powershell
# 1) 버전 올리고 커밋·푸시
#    src/NeisAutoFill.App/NeisAutoFill.App.csproj 의 <Version> 수정
git commit -am "chore: 버전 1.4.3로 올림"
git push origin main
# 2) 같은 버전으로 태그를 만들어 push  → 워크플로우가 나머지 전부 수행
git tag v1.4.3
git push origin v1.4.3
```
워크플로우가 하는 일: **① 태그와 csproj 버전 일치 확인 ② GAS_SECRET 존재 확인·주입 ③ 테스트 ④ self-contained 게시 ⑤ zip ⑥ 릴리스 생성**(노트 자동 생성).
버전 불일치나 시크릿 누락이면 게시 전에 실패하므로 **수동 릴리스의 대표 실수가 원천 차단**된다.

> ⚠ 자동 릴리스를 쓸 땐 수동 `gh release create` 를 쓰지 말 것 — 태그가 이중 생성돼 충돌한다. **태그 push 만** 한다.
> 진행 상황은 저장소 Actions 탭에서 확인. 노트를 다듬고 싶으면 게시 후 릴리스에서 편집.

## 3. 자체 포함(self-contained) 게시  *(수동 폴백)*

```powershell
dotnet publish src/NeisAutoFill.App -c Release -r win-x64 --self-contained true -o publish/NeisAutoFill
```

- 단일 exe·내부 압축·한국어 리소스 한정은 **csproj에 이미 박혀 있다** — 명령에 옵션을 더 붙일 필요 없음.
- 산출물 검증: `publish/NeisAutoFill/NeisAutoFill.App.exe` 의 ProductVersion이 올린 버전과 같은지 확인.

## 4. 압축

```powershell
Compress-Archive publish/NeisAutoFill/* publish/NeisAutoFill-win-x64.zip -Force
```

- 결과물 약 **97MB**. 에셋 이름은 반드시 `.zip` 으로 끝나야 한다(자동업데이터가 `.zip` 에셋을 찾음).
- zip 내부 구조는 **최상위 단일 폴더든 평면(파일이 루트)이든 둘 다** 자동업데이터가 처리한다(§6 참고).

## 5. 커밋 · 태그 · 릴리스  *(수동 폴백)*

```powershell
git add -A
git commit -m "chore: 버전 1.4.1로 올림"      # 커밋 메시지는 한글, Conventional Commits
git push origin main
gh release create v1.4.1 publish/NeisAutoFill-win-x64.zip --title "v1.4.1" --notes "..."
```

- 게시 후 확인: `gh release view v1.4.1 --json isDraft,assets` — `draft:false` + zip 에셋 첨부 확인.
- 커밋 규칙은 `README.md`(루트 마스터)와 전역 CLAUDE.md의 Conventional Commits 규약을 따른다(한글, 현재형).

## 6. 자동업데이트가 도는 방식 (`UpdateService.cs`)

| 단계 | 동작 |
|---|---|
| 감지 | 앱 시작 시 `releases/latest` 태그를 csproj `<Version>` 과 비교, 더 높으면 안내 |
| 다운로드 | 릴리스 에셋 중 이름이 `.zip` 으로 끝나는 것을 임시폴더로 받음 |
| 해제 | zip을 풀고, **최상위에 폴더 1개만 있으면 그 안**을, 아니면 푼 폴더 자체를 소스로 사용 |
| 검증 | 소스에 실행 파일(.exe)이 있는지 확인 — 없으면 중단(빈 소스로 미러 시 설치 폴더 삭제 방지) |
| 교체·재시작 | 앱 종료 대기 → **PowerShell 스크립트**가 `robocopy /MIR`로 미러 교체(옛 파일 정리) → 재시작. 경로 특수문자(%,&,^,')에 안전 |

→ 기존 사용자는 별도 조작 없이 다음 실행 때 업데이트 안내를 받는다.

## 7. 공개 저장소 제외(.gitignore) — 릴리스에 섞이면 안 되는 것

로컬 전용이라 저장소·릴리스에 올리지 않는다:

- `code.gs` — GAS 서버 원본 (배포는 Apps Script 콘솔에서, [GAS_서버.md](GAS_서버.md) 참고)
- `주소.txt` — GAS 웹앱 배포 주소
- `docs/archive/NEIS_자동입력기_개발노트.md` — Python 프로토타입 원형 노트
- `publish/` — 빌드 산출물

> GAS(code.gs)는 저장소에 없으므로 프로그램 릴리스와 **별개로** Apps Script에서 재배포해야 한다.
> 서버 쪽 절차는 [GAS_서버.md](GAS_서버.md) 참고.
